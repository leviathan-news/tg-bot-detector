## Project Overview

Heuristic bot detection toolkit for Telegram broadcast channel subscribers. Built on Telethon (MTProto). v1 is **analysis-only** — no destructive operations (banning/purging). The CLI generates scored candidate lists for offline human review.

Package name: `tg-bot-detector`, importable as `tg_purge`. CLI entry point: `tg-purge`.

## Commands

```bash
# Install (editable, for development)
pip install -e .

# Install with TOML config support on Python <3.11
pip install -e ".[toml]"

# Run tests
python -m pytest tests/ -v

# Run a single test file or test
python -m pytest tests/test_scoring.py -v
python -m pytest tests/test_scoring.py::TestScoreUser::test_deleted_user_scores_5 -v

# Run CLI
tg-purge analyze --channel @foo --strategy minimal
python -m tg_purge analyze --channel @foo
```

No linter or formatter is configured. No CI pipeline exists.
Requires Python >= 3.9.

### CLI subcommands

| Command | Purpose |
|---------|---------|
| `analyze` | Three-phase enumeration with score distribution and sample users |
| `candidates` | Export user IDs above threshold to CSV (with optional safelist) |
| `join-dates` | Join date distribution analysis (top days, hourly breakdown) |
| `spike` | Deep-dive analysis of a specific time window vs control group |
| `validate` | Score known-good users from CSV to measure false positive rate |
| `registry` | Local bot registry management: `generate`, `add`, `check` |

All subcommands accept `--delay` (override API query rate), `--session-path`, `--config`.

## Architecture

### Core flow

Every CLI command follows the same pattern:
1. `cli.py:main()` parses args via argparse, dispatches to `tg_purge/commands/<command>.py:run(args)`
2. Command loads config (`config.py:load_config`) — env vars override TOML override defaults
3. Command creates Telethon client (`client.py:create_client`) and resolves channel entity
4. Command calls enumeration functions to fetch subscribers via MTProto API
5. Command scores each user via `scoring.py:score_user` and formats output via `formatters.py`

### Scoring engine (`tg_purge/scoring.py`)

Central module. `score_user(user, config=None, join_date=None, spike_windows=None)` takes a Telethon User object, returns `(score: int, reasons: list[str])`. Higher score = more likely bot. Score is clamped to >= 0.

Key design decisions:
- Deleted accounts get +5 and early return (skip all other checks)
- `ScoringConfig` dataclass holds all weights — defaults are canonical, overridable for experimentation
- Status type detection uses `type(status).__name__` string comparison to avoid requiring Telethon at import time — this is intentional for testability
- Positive signals (premium, emoji_status) are stored as negative ints in config and added to score
- Optional `join_date` + `spike_windows` params enable spike detection signal (`spike_join`, +2) without breaking backward compatibility

### Enumeration (`tg_purge/enumeration.py`)

Two query sets: `MINIMAL_QUERIES` (22) and `FULL_QUERIES` (69, includes CJK chars 李/王). Recursive prefix expansion: when a query hits the 200-result API cap, sub-queries are generated by appending `a-z0-9` characters, up to configurable `max_depth` (default 3). This dramatically improves coverage on large channels.

`enumerate_subscribers()` uses a work-queue pattern and is the main entry point for `candidates`, `join-dates`, `spike`, and `registry generate`. The `analyze` command does its own phased enumeration (bots → recent → search) instead — it does **not** use recursive prefix expansion. Errors during individual queries are logged to stderr and skipped (non-fatal).

### Testing (`tests/`)

Tests use `MockUser` objects from `conftest.py` that mirror Telethon's `User` attribute interface. Mock status classes (`UserStatusEmpty`, `UserStatusOffline`, etc.) have matching `__name__` strings so `scoring.py`'s type-name-based dispatch works without Telethon installed. The `make_user` fixture is a factory with auto-incrementing IDs.

No integration tests exist — all tests are unit tests against scoring logic, CLI parsing, config loading, registry CRUD, and enumeration query sets/expansion logic. No Telegram API calls in tests. 130 tests total.

### Config priority

`config.py:load_config()`: Environment variables > TOML file > built-in defaults. Env var names: `TG_PURGE_API_ID`, `TG_PURGE_API_HASH`, `TG_PURGE_SESSION`, `TG_PURGE_CHANNEL`.

## Key Constraints

- Telegram's `GetParticipantsRequest` returns max 200 results per query — this is a hard API limit
- Search-based sampling is biased: accounts with no name or unusual names may be missed
- `no_status_ever` (+2) is the strongest single signal but also catches privacy-conscious real users who hide "last seen" — this is the primary source of false positives
- Score 2 is a cliff: 52.8% of sampled subscribers on the reference channel (@leviathan_news) scored exactly 2, making threshold ≥2 too aggressive for action
- The `score_user` function must remain importable and testable without Telethon installed

## Security-Sensitive Areas

- `*.session` files are Telegram login credentials — never commit, never log paths containing them. Session directory gets chmod 700, session file gets chmod 600 (best-effort)
- `registry/*.json` files contain real Telegram user IDs — gitignored, never distribute
- `config.toml` may contain API credentials — gitignored
- `client.py` prints connection info (user's first name, channel title, subscriber count) to **stderr** only — no PII on stdout
